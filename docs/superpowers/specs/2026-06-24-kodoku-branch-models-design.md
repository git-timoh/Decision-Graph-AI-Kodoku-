# Per-Branch Expand-Model Override — Design

**Goal:** Let a user optionally assign a different expand (reasoning) model to each top-level branch of a session at creation time, and tag every node with the model its branch runs on. Reframes the roadmap's "per-node model" (Phase D) as the simpler, more coherent per-branch override.

**Reframe:** Per-node was dropped — expansion runs in an automatic BFS loop, so there is no natural moment to pick a per-node model. A "branch" (a root candidate child and its subtree) is the natural unit, and config-time slot assignment works in autopilot and HITL alike.

## Scope

In scope:
- `SessionConfig.branch_models: list[str] | None` — optional, slot `i` maps to the root's `i`-th candidate branch.
- Override the **expand role only**. Evaluate and synthesize keep the session default so scoring stays comparable across branches.
- `Node.model: str | None` column tagging each node with its branch's expand model; inherited down the subtree.
- Frontend: per-branch model selects in the new-session dialog; a node model tag in the UI.

Out of scope (say so if wanted later):
- Per-branch evaluate/synthesize models.
- Reassigning a branch's model mid-run or at a checkpoint.
- Auto-cycling models across branches.

## Architecture

### 1. Config (`backend/kodoku/api/dtos.py`)
Add to `SessionConfig`:
```python
branch_models: list[str] | None = Field(default=None)
```
Validation (a `model_validator(mode="after")`, since it cross-checks `branching_factor`):
- `None` → no override (default behavior).
- `len(branch_models) <= branching_factor`.
- Each entry is either `""` (that slot uses the session default) or a valid LiteLLM model id — reuse the same `_MODEL_RE` check the existing `model` field uses.

`extra="forbid"` already set; default `None` means stored configs without the field still parse (backward compatible).

### 2. Node tag (`backend/kodoku/db/models.py`)
Add to `Node`:
```python
model: Mapped[str | None] = mapped_column(String, nullable=True)
```
Plus an additive alembic migration (down_revision = current head `a1b2c3d4e5f6`).

Semantics: `Node.model` = the expand model assigned to this node's branch (`null` = session default). Not "the model that generated this node's text" — the root's children are generated by the default expand model but tagged with their branch's model, because the tag answers "which model does this branch run on."

### 3. Client builder (`backend/kodoku/llm/factory.py`)
Add a helper that builds one client for an arbitrary model string, reusing BYOK key resolution:
```python
def build_client_for_model(model: str, settings: dict[str, str]) -> LLMClient
```
Refactor: `_build_client(role, settings)` already resolves model→provider→key→`LiteLLMClient`. Extract the model→client half into `build_client_for_model` and have `_build_client` call it with the role's resolved model. (Targeted refactor of code we're touching — the ollama base-url + key resolution logic moves with it unchanged.)

### 4. Engine wiring (`backend/kodoku/api/run.py`, `state_machine.py`)
- In `run.py`, after `make_role_clients`, build the override map from `config.branch_models`:
  `expand_overrides = {m: build_client_for_model(m, settings) for m in set(branch_models) if m}` (distinct, non-empty). Pass it into `DecisionEngine(... , expand_overrides=expand_overrides)`. Default `{}`.
- `DecisionEngine.__init__` gains `expand_overrides: dict[str, LLMClient] | None = None`, stored as `self._expand_overrides = expand_overrides or {}`.
- In `_expand_one(parent)`: pick the expand client —
  `client = self._expand_overrides.get(parent.model) or self.clients.expand` — and call `expand(client, ...)`. (Root's `model` is `None`, so the root always expands with the default.)
- When creating each child node, set `child.model`:
  - if `parent` is the root: `child.model = _slot_model(branch_models, index)` where `index` is the child's position in creation order, and `_slot_model` returns the slot string if non-empty else `None`.
  - else: `child.model = parent.model` (inherit down the subtree).
- Add `model` (the child's `model`) to the `node.created` event payload.
- Flush-only preserved; no new `commit()`.

### 5. Frontend
- `NewSessionDialog.tsx`: when `branching_factor` is chosen, render that many optional model `<select>`s (reuse `MODEL_PRESETS`), each defaulting to "" (= session default), collected into `branch_models` and sent in the config (omit / send `null` when all are empty).
- Node model tag: in `lib/ws/types.ts` add `model?: string | null` to `GraphNode`; the `node.created` reducer case already spreads the payload, so `model` flows in. Render a small model tag in the NodeDrawer (and/or on the graph node) when `model` is set.
- Regenerate `contracts.ts`.

## Data flow
Create session with `branch_models` → run.py builds `expand_overrides` (model→client) → engine expands root with the default model, tags each root child with its slot model, and each deeper node inherits its parent's tag → when expanding a tagged node, the engine looks up its override client (falls back to default) → `node.created` carries `model` → frontend shows the branch's model tag.

## Testing
- Config: `branch_models` validation — valid list, oversized list (> branching_factor) rejected, bad model id rejected, `""` slots allowed, `None` default.
- Factory: `build_client_for_model` returns a `LiteLLMClient` bound to the given model with the resolved key (no live call).
- Engine (fakes only): (a) a branch with an override model expands via the override client and its nodes are tagged; (b) slots left `""`/absent fall back to the default expand client and tag `None`; (c) descendants inherit the parent's `model`; (d) `branch_models=None` → behavior unchanged (existing suite is the regression guard), `node.created` carries `model: None`.
  Inject `expand_overrides` as a dict of `FakeLLMClient`s keyed by model string; assert which fake produced a branch's candidates (script distinct candidate titles per fake).
- Frontend: TS strict, typecheck + lint clean.

## Global constraints
- Python 3.12, `from __future__ import annotations`; mypy strict + ruff (E,F,I,B,UP,W) zero errors.
- Engine stays flush-only.
- Never hit a real provider in tests.
- TS strict, no `any`; regenerate `contracts.ts` after the dtos change.
- `branch_models=None` (default) → engine behavior byte-for-byte unchanged except `node.created` carrying the additive `model` field.
- Tests build schema via `Base.metadata.create_all`, so the new column appears automatically; the alembic migration is for real DBs.
